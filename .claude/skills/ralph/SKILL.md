# Ralph PRDコンバーター

既存のPRDをRalphが自律実行に使用するprd.json形式に変換します。

---

## 設計思想: テスト駆動と100%カバレッジ

**Ralphは100%テストカバレッジを目指す。** これは単なる数値目標ではなく、アーキテクチャの品質指標である。

### 核心原則

1. **純粋と不純の分離**: ビジネスロジック（純粋）とI/O・副作用（不純）を明確に分離する
2. **不純な領域の最小化**: 不純なコードは薄いレイヤーに押し込め、テスト対象外として明示的に除外する
3. **レイヤリングの徹底**: 依存関係は常に一方向。上位層は下位層に依存するが、逆は許さない
4. **テスタビリティ = 良い設計**: テストしづらいコードは設計が悪いサイン

### カバレッジ除外の対象（不純な領域）

以下はカバレッジ対象外として分離・抽象化する：

- I/O処理（ファイル、ネットワーク、DB接続の実体）
- フレームワーク固有のエントリーポイント（Next.jsのpage.tsx, route.tsなど）
- 環境依存の設定・初期化コード
- サードパーティライブラリのラッパー薄層

これらは**インターフェース/型で抽象化**し、純粋なロジックから分離する。

### アーキテクチャの階層例

```
[UI Components] ─依存→ [Application Services] ─依存→ [Domain Logic] ─依存→ [Repository Interface]
                                                                                    ↑
                                                                        [Repository実装(不純/除外)]
```

## やること

PRD（マークダウンファイルまたはテキスト）を受け取り、ralphディレクトリに`prd.json`として変換する。

---

## 出力フォーマット

```json
{
  "project": "[プロジェクト名]",
  "branchName": "ralph/[機能名-ケバブケース]",
  "description": "[PRDのタイトル/はじめにからの機能説明]",
  "userStories": [
    {
      "id": "US-001",
      "title": "[ストーリータイトル]",
      "description": "[ユーザー]として、[機能]を使って[メリット]を得たい",
      "acceptanceCriteria": ["基準1", "基準2", "型チェックが通る"],
      "priority": 1,
      "passes": false,
      "notes": ""
    }
  ]
}
```

## ストーリーサイズ: 最重要ルール

**各ストーリーは1回のRalphイテレーション（1つのコンテキストウィンドウ）で完了できなければならない。**

Ralphは各イテレーションごとに新しいAmpインスタンスを生成し、前の作業の記憶はない。ストーリーが大きすぎると、LLMは完了前にコンテキストを使い果たし、壊れたコードを生成する。

### 適切なサイズのストーリー:

- データベースカラムとマイグレーションの追加
- 既存ページへのUIコンポーネントの追加
- 新しいロジックでサーバーアクションを更新
- リストにフィルタードロップダウンを追加

### 大きすぎる（分割が必要）:

- 「ダッシュボード全体を構築」 - 分割: スキーマ、クエリ、UIコンポーネント、フィルター
- 「認証を追加」 - 分割: スキーマ、ミドルウェア、ログインUI、セッション処理
- 「APIをリファクタリング」 - エンドポイントまたはパターンごとに1ストーリーに分割

**経験則:** 変更を2〜3文で説明できない場合、それは大きすぎる。

---

## ストーリーの順序: 依存関係を先に

ストーリーは優先度順に実行される。前のストーリーは後のストーリーに依存してはならない。

**正しい順序:**

1. スキーマ/データベース変更（マイグレーション）
2. サーバーアクション/バックエンドロジック
3. バックエンドを使用するUIコンポーネント
4. データを集約するダッシュボード/サマリービュー

**間違った順序:**

1. UIコンポーネント（まだ存在しないスキーマに依存）
2. スキーマ変更

## 受け入れ基準: 検証可能でなければならない

各基準はRalphがチェックできるものでなければならず、曖昧であってはならない。

### 良い基準（検証可能）:

- "`status`カラムをtasksテーブルに追加、デフォルトは'pending'"
- "フィルタードロップダウンにオプション: すべて、アクティブ、完了"
- "削除をクリックすると確認ダイアログが表示される"
- "型チェックが通る"
- "テストが通る"

### 悪い基準（曖昧）:

- "正しく動作する"
- "ユーザーがXを簡単にできる"
- "良いUX"
- "エッジケースを処理する"

### 最後の基準として必ず含める:

```
"型チェックが通る"
"テストが通る"
"カバレッジが低下していない（または向上している）"
```

### UIを変更するストーリーの場合、これも含める:

```
"dev-browserスキルを使用してブラウザで確認"
```

フロントエンドのストーリーは視覚的に確認されるまで完了ではない。Ralphはdev-browserスキルを使用してページに移動し、UIを操作し、変更が機能することを確認する。

## ストーリー実行フロー: 前後確認の徹底

Ralphは各ストーリーの実行時に以下のフローを**必ず**守る：

### 📋 ストーリー開始前の確認

1. **テスト実行**: 現在のテストがすべて通ることを確認
2. **カバレッジ記録**: 現在のカバレッジ率を記録（ベースライン）
3. **型チェック**: `tsc --noEmit` または同等のコマンドで型エラーがないことを確認

```bash
# 例: 開始前の状態を記録
npm run test -- --coverage
# カバレッジ: 85.2% (statements)
```

### 🔨 実装中の原則

1. **テストファースト**: 可能な限り先にテストを書く
2. **純粋ロジック優先**: ビジネスロジックを純粋関数として抽出
3. **不純コードの分離**: I/O等は薄いアダプター層に閉じ込める
4. **小さなコミット単位**: 動く状態を保ちながら進める

### ✅ ストーリー完了前の確認

1. **テスト実行**: すべてのテストが通ることを確認
2. **カバレッジ比較**: ベースラインと比較し、低下していないことを確認
3. **型チェック**: 型エラーがないことを確認
4. **カバレッジ未達の場合**:
   - 不純な領域として除外すべきか検討
   - 除外する場合は明示的にカバレッジ除外設定に追加
   - 除外理由をコメントで記載

```bash
# 例: 完了前の確認
npm run test -- --coverage
# カバレッジ: 87.1% (statements) ✅ 向上

# または不純な領域を除外設定に追加
# coverage: { exclude: ['src/adapters/**'] }
```

### ⚠️ カバレッジが下がった場合

カバレッジ低下は**レッドフラグ**。以下を確認：

1. 新コードにテストを追加したか？
2. 不純な領域なら、適切に分離・除外設定されているか？
3. テスト困難なら、設計を見直すべきでは？

**カバレッジ低下を放置してストーリーを完了としてはならない。**

## 変換ルール

1. **各ユーザーストーリーは1つのJSONエントリになる**
2. **ID**: 連番（US-001, US-002, など）
3. **優先度**: 依存関係の順序に基づき、次にドキュメントの順序
4. **すべてのストーリー**: `passes: false` と空の `notes`
5. **branchName**: 機能名から派生、ケバブケース、`ralph/`プレフィックス
6. **必ず追加（すべてのストーリー）**:
   - 「型チェックが通る」
   - 「テストが通る」
   - 「カバレッジが低下していない」
7. **ロジックを含むストーリー**: テストを書く基準を明示的に含める
8. **不純な領域を扱うストーリー**: 「不純な領域として分離・抽象化されている」を含める

---

## 大きなPRDの分割

PRDに大きな機能がある場合は分割する：

**元:**

> 「ユーザー通知システムを追加」

**分割後:**

1. US-001: データベースに通知テーブルを追加
2. US-002: 通知送信用の通知サービスを作成
3. US-003: ヘッダーに通知ベルアイコンを追加
4. US-004: 通知ドロップダウンパネルを作成
5. US-005: 既読にする機能を追加
6. US-006: 通知設定ページを追加

各ストーリーは独立して完了・検証できる1つの集中した変更。

## 前回の実行のアーカイブ

**新しいprd.jsonを書き込む前に、別の機能からの既存のものがあるか確認する:**

1. 存在する場合は現在の`prd.json`を読む
2. `branchName`が新しい機能のブランチ名と異なるか確認
3. 異なっていて、かつ`progress.txt`にヘッダー以外のコンテンツがある場合:
   - アーカイブフォルダを作成: `archive/YYYY-MM-DD-feature-name/`
   - 現在の`prd.json`と`progress.txt`をアーカイブにコピー
   - `progress.txt`を新しいヘッダーでリセット

**ralph.shスクリプトは実行時にこれを自動的に処理する**が、実行間で手動でprd.jsonを更新する場合は、先にアーカイブすること。

## 保存前のチェックリスト

prd.jsonを書き込む前に確認:

### 基本チェック

- [ ] **前回の実行をアーカイブ**（異なるbranchNameのprd.jsonが存在する場合、先にアーカイブ）
- [ ] 各ストーリーは1イテレーションで完了可能（十分に小さい）
- [ ] ストーリーは依存関係順に並んでいる（スキーマ→バックエンド→UI）
- [ ] 受け入れ基準は検証可能（曖昧でない）
- [ ] どのストーリーも後のストーリーに依存していない

### テスト・カバレッジチェック

- [ ] すべてのストーリーに以下の基準が含まれている:
  - 「型チェックが通る」
  - 「テストが通る」
  - 「カバレッジが低下していない」
- [ ] ロジックを含むストーリーには、そのロジックのテストを書く基準が含まれている
- [ ] 不純な領域（I/O、外部API等）を扱うストーリーには、分離・抽象化の基準が含まれている

### UI・ブラウザチェック

- [ ] UIストーリーに「dev-browserスキルを使用してブラウザで確認」が基準として含まれている

### アーキテクチャチェック

- [ ] 純粋なロジックと不純な領域が明確に分離されるよう設計されている
- [ ] 依存関係が一方向になるよう階層化されている
- [ ] テスト困難なコードが最小化されている
