# 差し込みタスク

- [ ] 証明完了判定を厳密化する。現状は許可されていない公理が使われていてもproof completedとマークされてしまう。
    - クエストにおいては、各ゴールが使ってよい公理が定義されている前提とする。その仕組みがなければ、まずはそこから構築する。
    - rootノードはすべて、公理スキーマそのもの → 代入操作ノード → 具体的な式、という形で繋がっているべき。直接代入後の式を公理として置くのではなく、代入ステップを明示する。
    - 使ってはいけない公理がルートに含まれている場合、proof completedとしてマークしてはならない。
    - 参考: http://100.73.163.99:13006/?path=/story/proofpad-proofflowdemo--identity-proof-completed
    - 関連するデモストーリーをこの方針に沿って更新する
    - 関連するストーリー・テストの更新
- [ ] 設計がミスっていそう。MP適用があるものについて、そのコネクションを消したら、もう片方の適用される側などもコネクションが消えて、ノードはderivedではなく、浮いてるノードになるべきだ。
    公理としてマークできる機能はいらない気がしてきた。
- [ ] ゴールも、ゴールノードではなく、サイドパネルや一覧が提示されていて、それをキャンバスのどこかで、（使っていい公理だけで）証明されればゴールということにして、公理だとかゴールだとか言う内部ステートもやめよう。
- [ ] derivedというのも、コネクションの結果としてそのようにcomputedなものとして見れるというだけで、内部的にそれを管理する必要はない。ノードそのものは単なるformulaでしかなく。コネクションの状況でderivedになるだけだ、としよう。
- [ ] 同一のノードをマージできるようにしよう。このとき、上からのコネクション（derive）を持つもの同士はどちらから引っ張ればいいか自明でないので、先に選択された方など、リーダー的なのを決めて、そちらからderiveされたことにしよう。逆に、定理として利用されてる立場のコネクションはどちらも保持され、正しくマージされるべきだろう。
